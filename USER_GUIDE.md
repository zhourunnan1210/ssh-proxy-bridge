# SSH Proxy Bridge 使用手册

SSH Proxy Bridge 用于把 Windows 本机代理通过 SSH 安全地提供给远程 Linux 服务器，并在 VS Code 中直接打开指定的服务器目录。服务器上的 Codex、终端和其他命令行工具因此可以使用 Windows 代理访问网络。

## 使用前准备

开始前请确认：

- Windows 代理软件已经启动，并记住它的监听端口，例如 `7897`。
- Windows 已安装 OpenSSH Client。
- 已安装 VS Code、Remote - SSH 扩展和 Codex 扩展。
- 服务器允许 SSH 登录和 TCP 端口转发。普通服务器建议允许公钥认证；只提供密码认证的云平台 SSH 网关也可以使用。
- 你知道服务器的 SSH 地址、端口、用户名和密码。

## 启动应用

无论是当前开发目录还是解压后的便携包，日常只双击根目录中的：

```text
SshProxyBridge.exe
```

如果刚拉取源码、根目录还没有这个 EXE，开发者只需运行一次：

```text
start-ssh-proxy-bridge-gui.cmd
```

该脚本会生成最新自包含版本并启动同一个根目录 EXE。不要直接运行 `app`、`bin`、`obj` 或 `release` 子目录里的程序；这些都是构建过程使用的中间文件或分发副本。应用已经运行时不会重复启动。

应用采用单窗口操作：添加、编辑、删除服务器，输入密码，确认 SSH 主机指纹以及查看成功或错误提示，都会在当前主窗口内展开。操作面板显示时主页面会高斯模糊，完成或取消后会原位恢复并返回服务器页面，不会再弹出独立窗口。添加服务器向导会跟随主窗口高度调整，中间表单可滚动，底部操作按钮始终保留。

左下角“本机代理”显示当前服务器 Profile 配置的代理地址：

| 标记 | 含义 |
|---|---|
| 绿色 | 代理端口正在监听，可以继续连接。 |
| 灰色 | 代理端口暂时无法连接，请检查代理软件和端口。 |

切换服务器时，应用会自动重新检测对应的本机代理端口。

## 使用已有的当前连接

如果列表中存在带有“当前连接”标记的服务器，它来自现有 `config.local.json` 配置。

1. 启动 Windows 代理软件。
2. 启动 SSH Proxy Bridge。
3. 选择“当前连接”。
4. 确认左下角本机代理为绿色。
5. 点击“连接并打开 VS Code”。

应用不会在启动时自动停止或重建已有隧道。只有明确点击“停止连接”才会停止所选 Profile 的受管隧道。

## 添加一台新服务器

点击右上角“添加服务器”，主窗口会原位显示三步向导。按照提示填写信息；需要退出时点击“取消”即可返回服务器页面。

### 第一步：SSH 服务器

- 服务器名称：便于识别的名称，例如“GPU 服务器”。
- SSH 地址：服务器 IP 或域名。
- SSH 端口：默认通常为 `22`，以服务器提供的信息为准。
- SSH 用户：用于远程开发的 Linux 用户。
- 服务器密码：用于验证身份和初始化 SSH 登录。
- 保存密码：勾选后，认证成功的密码保存到 Windows Credential Manager，不会写入普通文件。普通公钥服务器可以不保存；如果云平台网关只允许密码认证，则必须勾选，软件会在初始化时明确提示。

应用会显示服务器 SSH 主机密钥类型和 SHA256 指纹。应当通过云服务商控制台或服务器管理员提供的信息核对指纹；无法确认时不要继续。

### 第二步：Windows 本机代理

- 代理地址通常填写 `127.0.0.1`。
- 代理端口填写代理软件实际监听端口。
- 不确定协议时选择“自动检测”。
- 可以选填代理程序路径，以便代理未运行时尝试启动。

### 第三步：项目与安全选项

- 默认远程目录必须是 Linux 绝对路径，例如 `/workspace/project`。
- 服务器代理端口通常保留默认值。端口被占用时，初始化会在安全范围内自动选择其他端口。
- 普通使用建议选择“一键模式”。应用会为该 Profile 创建独立 ED25519 Key，并通过当前 Windows 用户的 NTFS 权限保护私钥。

完成密码和主机指纹验证后，服务器状态显示为“密码已验证”。

## 完成 SSH 初始化

选择“密码已验证”的服务器，然后点击“完成 SSH 初始化”。应用会先读取服务器公布的认证方式，然后自动选择下面一种模式：

- **普通服务器（Key 模式）**：生成独立 SSH Key、幂等安装公钥，并验证 Key-only 登录。
- **密码网关模式**：适用于 SSHPiper、云平台统一登录网关等只公布 `password` 的服务。软件不再反复安装无效公钥，而是通过受控 AskPass Helper 从 Windows Credential Manager 按需读取密码。

随后应用会检查服务器代理端口、自动避开已占用端口，并生成应用专属 `known_hosts`、SSH Config 和不含明文密码的运行配置。

全部成功后，状态变为“可以连接”。两种模式都固定使用首次确认的 SSH 主机指纹。Key 模式重复初始化不会重复添加相同公钥。

## 日常连接并打开 VS Code

1. 启动 Windows 代理软件。
2. 启动 SSH Proxy Bridge。
3. 选择目标服务器。
4. 确认本机代理圆点为绿色。
5. 点击“连接并打开 VS Code”。

应用会建立 SSH 反向隧道、从服务器验证代理可用性、更新受管的远程代理环境，然后让 VS Code Remote-SSH 打开该 Profile 保存的远程目录。

如果隧道已经健康运行，重复点击不会叠加新的受管隧道进程。

VS Code 启动后，应用会结束“处理中”状态并显示“已连接”。VS Code 作为独立程序继续运行，不需要等到关闭 VS Code 后应用才完成连接流程。

应用的一次性 SSH 检查不会继承用户 SSH Config 中的 `LocalForward`、`RemoteForward` 或 `DynamicForward`，因此已有的其他转发不会让状态检查或连接流程一直等待。单次 SSH 检查超过 20–35 秒会安全失败并显示原因，GUI 自身也有整体等待上限。

## 当前服务器操作

| 操作 | 说明 |
|---|---|
| 运行诊断 | 检查本机代理、OpenSSH、SSH 认证、隧道和服务器代理。 |
| 刷新状态 | 读取本机代理、隧道和 SSH 登录状态。 |
| 停止连接 | 只停止当前所选 Profile 的受管隧道。 |
| 连接并打开 VS Code | 建立或复用隧道并打开默认远程目录。 |

## 编辑服务器

“编辑”目前可以修改：

- 服务器显示名称。
- 本机代理地址、端口和协议。
- 代理程序路径和自动启动选项。
- VS Code 默认远程目录。

编辑不会改变 SSH 地址、SSH 用户、主机指纹或专用 Key。代理设置改变后，应先停止该 Profile，再重新连接。

## 删除服务器

“删除”只适用于应用创建的 Profile，不会删除“当前连接”的旧配置。

删除 Ready Profile 前，应用会先停止它自己的受管隧道。你可以选择同时删除：

- Windows Credential Manager 中保存的服务器密码。
- 本机 SSH Config 托管块。
- 该 Profile 的专用 SSH Key。

本地运行配置、`known_hosts` 和状态日志会随 Profile 删除。为避免破坏仍可使用的登录方式，服务器 `authorized_keys` 和远程 Shell 代理块目前默认保留。

## 常见问题

### 本机代理圆点是灰色

1. 确认代理软件已经启动。
2. 在服务器编辑页面核对代理地址和端口。
3. 切换一次服务器或点击“刷新状态”。
4. 如果仍为灰色，确认代理软件确实监听 `127.0.0.1`，并检查防火墙或代理软件设置。

### 本机代理是绿色，但服务器代理失败

运行诊断并查看失败层级。常见原因包括服务器禁止 TCP 转发、远端代理端口被其他程序占用、SSH 隧道退出，或者本机代理协议配置不正确。

### Codex 反复提示 `stream disconnected before completion`

这通常表示 SSH 隧道已因网络切换、电脑休眠或连接重置而退出，不一定是服务器或代理软件损坏。

1. 回到 SSH Proxy Bridge，点击“刷新状态”。
2. 如果显示隧道已停止，点击“连接并打开 VS Code”。软件会清理失效的隧道状态并重新建立连接。
3. 如果仍不能使用，点击“运行诊断”。诊断会分别检查本机代理、SSH 登录、受管隧道，以及服务器能否通过隧道访问公网。
4. 看到“服务器可以通过 Windows 代理隧道访问公网”后，回到 VS Code 重试 Codex；通常不需要重启服务器。

联网诊断会依次尝试多个相互独立的测试地址，单个网站临时返回 502 不会再被直接判定为整个代理失效。

### SSH Key 登录失败

确认服务器没有删除对应的 `authorized_keys` 项，并检查服务器是否允许公钥认证。必要时重新添加 Profile 或使用后续版本提供的“修复公钥”功能。

### 提示 `No suitable authentication method ... (password)`

这通常说明连接的是只允许密码认证的云平台 SSH 网关，而不是用户名写错。新版会在“完成 SSH 初始化”时自动切换到密码网关模式。请确保添加服务器时勾选“保存密码”，然后重新完成初始化。密码只保存在 Windows Credential Manager；如果 VS Code Remote-SSH 自己仍显示密码输入框，按正常方式输入一次即可。

### VS Code 提示“扩展在磁盘上已被修改，请重新加载窗口”

这通常是 VS Code 或扩展更新后的正常提示，不代表代理隧道故障。保存工作后点击“重新加载窗口”即可。

### 打开的远程目录不正确

点击“编辑”，把默认远程目录改为以 `/` 开头的正确 Linux 绝对路径，然后重新连接。

## 安全说明

- 服务器密码不会进入 Profile JSON、运行配置、SSH Config、日志或命令行。
- 保存的密码位于当前 Windows 用户的 Credential Manager。
- 密码网关模式使用受控 AskPass Helper，经标准输入输出把凭据只交给当前 SSH 子进程；密码本身不会进入环境变量或临时文件。
- 首次连接必须人工确认 SSH SHA256 主机指纹。
- 后续连接固定使用已确认的主机密钥，并启用严格主机密钥检查。
- 反向代理默认只绑定服务器 `127.0.0.1`，不会直接暴露到服务器外部网络。
- 一键模式私钥没有口令，安全性依赖 Windows 账号和 NTFS ACL；多人共用 Windows 账号时不建议使用一键模式。

## 从旧版本升级

新版数据位于 `%LOCALAPPDATA%\SshProxyBridge`。首次启动时，应用会把旧 `%LOCALAPPDATA%\CodexRemoteBridge` 中尚不存在的新文件复制过去，但不会删除旧目录、覆盖新文件或停止正在运行的 SSH 隧道。旧版 Windows 凭据会在首次使用时复制到新凭据名称，旧凭据仍会保留。

旧 `config.local.json`、`.state`、SSH Config 托管块和远端 Shell 代理块继续兼容。确认新版本稳定前，不需要手工删除任何旧文件。

## 项目声明

SSH Proxy Bridge 是独立的开源项目，与 Microsoft 或 OpenAI 没有关联，也未获得其认可或背书。

## 当前限制

- 带口令的 SSH 私钥尚不能自动初始化；当前支持一键无口令 Key 和密码网关两种模式。
- 修改 SSH 地址、用户或主机指纹时，需要重新创建 Profile。
- 删除 Profile 暂不自动删除服务器端公钥和远程 Shell 配置。
- 隧道停止后若端口后来被其他程序占用，当前版本会安全失败，但尚不会在每次日常连接时自动改写端口。
- 应用关闭后不会常驻后台自动重连；网络中断后请重新打开应用并点击“连接并打开 VS Code”。
